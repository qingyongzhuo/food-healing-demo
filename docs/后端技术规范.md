## 1. 总则

### 1.1 规范目的

- 统一团队代码风格与工程行为，降低协作摩擦
- 保证生产环境可部署、可观测、可回滚
- 让新成员"克隆仓库 → `uv sync` → `uv run uvicorn ...`"即可跑起来
- 为 AI 双平台调用、双数据源、配置中心等复杂场景提供明确边界

### 1.2 适用范围

- `backend/` 目录下所有 Python 源码、配置、脚本、测试
- 后端组所有成员的本地开发、CI、生产部署行为
- 与前端、运维、AI 平台对接的接口契约

### 1.3 Python 版本

- **统一使用 Python 3.11+**（推荐 3.12）
- 理由：`asyncio.TaskGroup`、`typing.Self`、PEP 695 类型语法（可选）、异常 group 等特性
- `pyproject.toml` 中 `requires-python = ">=3.11"`
- CI 与生产服务器 Python 版本必须一致（由 `uv` 在 `.python-version` 中固定）

### 1.4 规范原则

1. **可执行优先**：每条规范必须能通过 `ruff`、`mypy`、`pytest` 或人工 review 验证
2. **最小改动优先**：规范增量与现有 V2 设计对齐，不推翻已有架构
3. **基础设施合理用**：Pg/Redis/MQ/OSS/Nacos 各司其职，不堆砌
4. **安全默认**：所有外部输入默认不信任，所有密钥默认不入库

## 2. uv 项目管理规范（重点章节）

### 2.1 为什么用 uv

- **速度**：uv（Astral 出品，Rust 实现）比 pip 快 10–100x，本地 `uv sync` 通常 <5s
- **确定性**：`uv.lock` 锁定全部依赖（含传递依赖）的精确版本与 hash，跨机器复现一致
- **统一工具链**：替代 `pip + virtualenv + pip-tools + pyenv` 多套工具
- **PEP 621 标准**：用 `pyproject.toml` 作为项目元数据单一来源
- **兼容性**：可 `uv export` 导出 `requirements.txt`，兼容 Docker / 老脚本

### 2.2 安装 uv

```bash
# macOS / Linux
curl -LsSf https://astral.sh/uv/install.sh | sh

# Windows (PowerShell)
powershell -ExecutionPolicy ByPass -c "irm https://astral.sh/uv/install.ps1 | iex"

# 验证
uv --version
```

### 2.4 uv.lock 锁定规范

- **必须提交 `uv.lock` 到 git**（即使文件较大），它是跨机器复现的关键
- **禁止手工编辑 `uv.lock`**，所有变更必须通过 `uv add` / `uv remove` / `uv lock` 触发
- 升级依赖：`uv lock --upgrade`（升级所有允许范围内的依赖）或 `uv lock --upgrade-package <name>`（只升一个）
- CI 必须校验 `uv.lock` 与 `pyproject.toml` 一致：`uv lock --check`

### 2.5 .python-version 文件

`backend/.python-version` 内容固定为：

```
3.12
```

- `uv` 会自动读取此文件，安装并使用对应 Python 版本
- 团队成员无需手动 `pyenv install`，`uv sync` 自动处理

### 2.6 常用命令速查

| 场景 | 命令 |
|---|---|
| 首次拉代码后安装 | `uv sync`（自动创建 .venv + 装好全部依赖） |
| 安装含 dev 依赖 | `uv sync --extra dev` |
| 添加运行时依赖 | `uv add "fastapi>=0.115,<0.116"` |
| 添加 dev 依赖 | `uv add --dev pytest` |
| 移除依赖 | `uv remove aio-pika` |
| 升级所有依赖 | `uv lock --upgrade` |
| 升级单个依赖 | `uv lock --upgrade-package openai` |
| 在虚拟环境中运行 | `uv run uvicorn app.main:app --reload --port 8000` |
| 在虚拟环境中跑测试 | `uv run pytest` |
| 在虚拟环境中跑脚本 | `uv run python scripts/seed_foods.py` |
| 导出 requirements.txt（兼容老脚本/Docker） | `uv export --format requirements-txt --output requirements.txt` |
| 导出不含 dev 依赖 | `uv export --no-dev --format requirements-txt -o requirements.txt` |
| 锁文件一致性校验 | `uv lock --check` |

### 2.7 虚拟环境管理

- `uv` 自动在 `backend/.venv` 创建并管理虚拟环境，**禁止手工 `python -m venv`**
- `.venv` 必须加入 `.gitignore`
- 激活方式（可选）：`source .venv/bin/activate`（Linux）或 `.venv\Scripts\activate`（Windows）
- 不激活也能用：`uv run <cmd>` 自动使用项目虚拟环境

### 2.8 与 requirements.txt 的关系

- **`pyproject.toml` + `uv.lock` 是唯一真实来源**
- `requirements.txt` 仅作为**导出产物**，用于：
  - Docker 镜像构建（`uv export --no-dev -o requirements.txt` 后 `pip install -r`）
  - 兼容不支持 uv 的旧 CI / 部署脚本
- **禁止手工编辑 `requirements.txt`**
- 导出命令加入 `Makefile` 或 CI 脚本，每次发版前重新生成

### 2.9 团队协作流程

**新成员入门**：

```bash
git clone <repo>
cd food-healing-demo/backend
uv sync --extra dev         # 装好全部依赖
cp .env.example .env       # 填入双 API Key 与连接配置
uv run uvicorn app.main:app --reload --port 8000
```

**新增依赖流程**：

1. 评估必要性（是否可用现有依赖替代）
2. `uv add <package>`（自动更新 `pyproject.toml` + `uv.lock`）
3. 本地 `uv run pytest` 验证
4. 提交 `pyproject.toml` + `uv.lock`
5. PR 中说明新增理由

**升级依赖流程**：

1. `uv lock --upgrade-package <name>` 或 `uv lock --upgrade`（全量）
2. `uv run pytest` 跑全量测试
3. 本地手动验证关键路径（识图、对话、周报）
4. 提交 `uv.lock`

**禁止行为**：

- 禁止 `pip install` 直接装包（绕过 uv，破坏 lockfile 一致性）
- 禁止手工修改 `uv.lock`
- 禁止提交 `.venv` 目录
- 禁止提交 `.env`（含密钥）

## 4. 代码规范

### 4.1 命名规范

参见 §3.3，并补充：

- 缩写保持大小写一致：`userId`（API 字段 camelCase）vs `user_id`（Python 变量 snake_case）
- API 字段命名与前端 `foodTray` / `addFood` 入参完全对齐（见 V2 设计 §13.4）
- 布尔变量以 `is_` / `has_` / `should_` 开头：`is_custom`、`has_allergen`
- 避免单字母变量，循环计数器 `i/j/k` 除外

### 4.2 类型注解规范

- **所有函数签名必须带类型注解**（ruff `ANN` 规则强制）
- 用 Python 3.11+ 原生语法：`list[str]` 而非 `List[str]`，`dict[str, Any]` 而非 `Dict[str, Any]`
- 可选类型用 `X | None`，不用 `Optional[X]`
- Pydantic 模型字段必须注解
- `Any` 谨慎使用，能用更具体类型就用

```python
# 正确
async def recognize_dish(
    file: UploadFile,
    user_id: str | None = None,
    meal_hint: Literal["breakfast", "lunch", "dinner"] | None = None,
) -> RecognizeTask: ...

# 错误（无注解）
async def recognize_dish(file, user_id=None, meal_hint=None): ...
```

### 4.3 注释与 docstring 规范

- **Google 风格** docstring，所有公开函数 / 类必须有
- 模块顶部 docstring 说明职责

```python
def build_chat_system_prompt(habits: UserHabits, history: list[dict]) -> str:
    """组装对话 system prompt，注入用户习惯 + 短期记忆。

    Args:
        habits: 用户长期习惯，来自 MySQL user_habits 表。
        history: 最近 N 轮对话，来自 Redis chat:ctx:{userId}。

    Returns:
        拼装好的 system prompt 字符串。

    Raises:
        ValueError: 当 habits.userId 为空时。
    """
    ...
```

- 行内注释只解释"为什么"，不解释"是什么"
- TODO 必须带署名与 issue 号：`# TODO(@zhang, #123): 改成异步批量`

### 4.4 异步规范

- **路由、service、repo 层默认 async**
- 禁止在 async 函数中调用同步阻塞 IO（`requests.get`、`time.sleep`、`open()`）
- 阻塞操作用 `asyncio.to_thread()` 包装：
  ```python
  img = await asyncio.to_thread(Pillow.open, path)
  ```
- AI 客户端调用必须 async（dashscope 与 openai SDK 均支持）
- DB 连接用 async 驱动（aiomysql / asyncpg / redis.asyncio）
- `asyncio.gather` 并发独立任务，`asyncio.Semaphore` 控制并发上限

### 4.5 Lint 与格式化

- 用 `ruff format` 格式化（替代 black）
- 用 `ruff check` 检查代码质量
- 用 `mypy` 静态类型检查
- CI 必须三关全绿才能合并

```bash
uv run ruff format .
uv run ruff check .
uv run mypy app/
```

## 5. FastAPI 规范

### 5.1 路由组织

- 每个业务模块一个 `APIRouter`，文件放 `routes/`
- `main.py` 集中注册，前缀统一 `/api`

```python
# app/routes/recognize.py
from fastapi import APIRouter, Depends
from app.deps import get_user_id

router = APIRouter(prefix="/recognize", tags=["recognize"])

@router.post("/recognize-dish")
async def recognize(...): ...
```

```python
# app/main.py
from fastapi import FastAPI
from app.routes import recognize, chat, recommend, ...

app = FastAPI(title="食愈校园", version="2.0.0")
app.include_router(recognize.router, prefix="/api")
app.include_router(chat.router, prefix="/api")
# ...
```

### 5.2 依赖注入

- DB session、Redis、user_id 等通过 `Depends` 注入
- 公共依赖放 `app/deps.py`，模块特有依赖放模块内

```python
# app/deps.py
from typing import Annotated
from fastapi import Header, HTTPException

async def get_user_id(x_user_id: Annotated[str | None, Header()] = None) -> str:
    if not x_user_id:
        raise HTTPException(status_code=401, detail={"code": 40101, "message": "未鉴权"})
    return x_user_id

UserIdDep = Annotated[str, Depends(get_user_id)]
```

- 鉴权当前用简易 header（P0），后续可扩展 JWT
- 不要在路由函数里直接 `import` repo / service 之外的内部模块

### 5.3 中间件

按以下顺序挂载（顺序重要）：

1. **request_id 注入**（最先，后续日志可用）
2. **CORS**
3. **请求日志**（记录入参、耗时）
4. **限流**（slowapi）
5. **全局异常处理**（最后兜底）

```python
# app/main.py
from app.middlewares.request_log import RequestLogMiddleware
from app.middlewares.error_handler import register_exception_handlers

app.middleware("http")(RequestLogMiddleware())
app.add_middleware(CORSMiddleware, ...)
register_exception_handlers(app)
```

### 5.4 Pydantic schema 规范

- 请求与响应模型分离，不复用同一个
- 请求模型后缀 `Request`，响应模型后缀 `Response`
- 内部模型无后缀
- 用 `model_config = ConfigDict(extra="forbid")` 拒绝多余字段

```python
# app/schemas/chat.py
class ChatRequest(BaseModel):
    model_config = ConfigDict(extra="forbid")
    userId: str
    message: str
    mode: Literal["daily","fitness","weight_loss"] = "daily"
    tray: list[TrayItem] = []

class ChatResponse(BaseModel):
    text: str
```

## 6. 数据库规范

### 6.1 PostgreSQL 访问层（关系型主库）

**驱动**：`asyncpg`（异步）+ `SQLAlchemy 2.0 async`

本项目使用 PostgreSQL 作为唯一关系型数据库，共 8 张表（定义在 `app/models/pg_orm.py`）：

| 表 | 用途 |
|---|---|
| `user` | 用户基础信息（昵称/密码hash/手机号/主题/状态） |
| `user_body_target` | 身体数据 & 营养目标（身高/体重/性别/年龄/目标类型/三大营养素） |
| `standard_food` | 系统标准食材库（6 大分类，每 100g 营养值） |
| `user_custom_food` | 用户自定义食材 |
| `user_collect_food` | 用户收藏食材关联 |
| `daily_diet_record` | 每日饮食总记录 |
| `meal_item` | 单餐食物明细 |
| `system_message` | 系统消息（饮食提醒/AI 推送） |

**session 管理**：

```python
# app/database.py
from sqlalchemy.ext.asyncio import create_async_engine, async_sessionmaker

engine = create_async_engine(settings.PG_DSN, pool_size=10, max_overflow=5)
AsyncSessionLocal = async_sessionmaker(engine, expire_on_commit=False)
```

- **禁止在路由层直接拿 session 写 SQL**，必须经 services 层
- 写操作用 `session.begin()` 自动提交
- 时序数据（营养记录）用 PostgreSQL `JSONB` 字段存储

### 6.2 MongoDB 访问层（文档存储，可选）

**驱动**：`motor`（异步 MongoDB 驱动）

MongoDB 用于存储非结构化数据，未配置时 graceful 降级：

| 集合 | 用途 |
|---|---|
| `ai_chat_history` | AI 对话历史（按 user_id + record_date 唯一键） |
| `camera_recognize_log` | 拍照识菜日志 |
| `ai_daily_report` | AI 每日营养报告 |

- `MONGO_URL` 为空时所有 MongoDB 操作 graceful 跳过，不阻塞功能

### 6.3 Redis 访问层

**驱动**：`redis.asyncio` + `hiredis` 解析器

**Key 命名规范**（统一前缀 `food_healing:`）：

| Key 模式 | 类型 | TTL | 用途 |
|---|---|---|---|
| `food_healing:recognize:{image_hash}` | String(JSON) | 7d | 识图结果缓存 |
| `food_healing:chat:ctx:{userId}` | List | 1h | 对话短期记忆 |
| `food_healing:dish:heat:{week}` | ZSET | 7d | 热门菜品排行 |
| `food_healing:ratelimit:{ip}:{route}` | String | 60s | 限流计数 |
| `food_healing:menu:{canteen}:{date}` | String(JSON) | 1h | 今日菜单缓存 |
| `food_healing:leaderboard:{mode}` | ZSET | 1h | 匿名榜缓存 |
| `food_healing:task:recognize:{task_id}` | String(JSON) | 30m | 异步任务状态 |

- **所有 key 必须带前缀**，禁止裸 key
- **所有 key 必须设 TTL**，禁止永久 key（除非字典数据，主动失效）
- 连接池复用，禁止每次请求 `Redis()`

```python
# app/data/redis_client.py
import redis.asyncio as redis
from app.config import settings

pool = redis.ConnectionPool.from_url(settings.REDIS_URL, max_connections=50)
redis_client = redis.Redis(connection_pool=pool)
```

### 6.4 迁移规范（Alembic）

- 用 **Alembic** 管理 PostgreSQL schema 变更
- **禁止手工改表结构**，所有变更走迁移
- 命名规范：`<日期>_<简述>.py`，如 `2026_07_03_add_user_habits.py`
- 迁移必须可回滚（`upgrade()` + `downgrade()`）
- 发版前在 staging 跑 `alembic upgrade head` 验证

```bash

# 升级

# 回滚
```

### 7.3 prompt 模板管理

- 模板定义在 `app/ai/prompts.py`（本地 fallback）
- **Nacos 动态下发优先**，启动时拉取 + 监听变更
- Nacos 配置 dataId 命名：`food_healing.prompt.{scene}`
- 监听变更后覆盖 `app/ai/prompts.py` 内存缓存（不写文件）

```python
# app/ai/prompts.py
class PromptManager:
    def __init__(self):
        self._cache: dict[str, str] = {}

    def get(self, scene: str) -> str:
        return self._cache.get(scene) or LOCAL_FALLBACKS[scene]

    def update(self, scene: str, content: str) -> None:
        self._cache[scene] = content
```

- 模板用 `str.format()` 或 `string.Template` 占位符，禁止 f-string（避免注入）
- 占位符规范：`{habits}` / `{history}` / `{nutrition}` / `{aiPersonality}`

### 7.4 用户习惯拼装 system prompt 规范

`ChatService` 拼装流程（见 V2 设计 §8.2）：

1. 取长期记忆：MySQL `user_habits`（`habit_repo.get(user_id)`）
2. 取短期记忆：Redis `food_healing:chat:ctx:{userId}`（最近 4 轮）
3. 取当前上下文：请求中的 `tray` + `nutrition` + `message`
4. 套 `CHAT_SYSTEM_TEMPLATE` 模板，填充占位符
5. system prompt 总长度 < 800 token，超长则截断短期记忆

- 拼装函数放 `chat_service._build_system_prompt()`，私有
- 单元测试覆盖：空 habits、空 history、超长 history 三种场景

### 7.5 token 控制

| 场景 | 输入 token 上限 | max_tokens |
|---|---|---|
| 识菜 | 图片 token 由分辨率决定，预处理缩放控制 | 300 |
| 对话 | system + history < 1500 token | 512 |
| 周报 | 7 条 DailySummary < 500 token | 300 |
| 情绪治愈 | < 200 token | 80 |
| 摘要 | 旧摘要 + 6 轮 < 800 token | 150 |

### 7.6 超时重试与降级策略

- 客户端配置 `timeout` 与 `retry`（见 §7.2 路由表）
- 重试条件：网络错误、5xx、429
- 重试间隔：指数退避 1s、2s
- 降级链：主模型 → fallback 模型 → 本地兜底（如 `generateAiText()` 复刻版）
- 降级触发后必须打 `WARN` 日志，方便监控

## 8. 异步任务规范

### 8.1 MQ 消费者

- 用 **aio-pika**（RabbitMQ 异步客户端）
- 每个 worker 独立进程，`uv run python -m app.workers.recognize_worker`
- 消费者放 `app/workers/`，命名 `{scene}_worker.py`
- 消费失败的消息进死信队列（DLX），不无限重试

### 8.2 task_id 设计

- 格式：`task_{ulid}`（ULID 全局唯一、有序）
- 任务状态存 Redis：`food_healing:task:recognize:{task_id}`，TTL 30 分钟
- 状态机：`pending` → `processing` → `done` / `failed`

```python
{
    "taskId": "task_01HXXX...",
    "status": "pending",
    "result": null,
    "createdAt": "2026-07-03T14:30:00+08:00",
    "error": null
}
```

### 8.3 轮询接口

`GET /api/recognize/result/{task_id}`：

- 读 Redis，命中直接返回
- 未命中（超过 30 分钟 TTL）返回 `40801` 识图任务超时

### 8.4 超时清理

- Redis TTL 自动清理 task 状态
- MQ 消息设 `expiration`（识图 60s、周报 5min），超时进 DLX
- 定时任务（cron 或 MQ 定时消息）每小时扫一次 DLX，告警 + 落库

### 10.3 异常中间件

```python
# app/middlewares/error_handler.py
from fastapi import Request
from fastapi.responses import JSONResponse
from app.schemas.common import ApiResponse

class BizError(Exception):
    def __init__(self, code: int, message: str, http_status: int = 400):
        self.code = code
        self.message = message
        self.http_status = http_status

def register_exception_handlers(app):
    @app.exception_handler(BizError)
    async def biz_handler(request: Request, exc: BizError):
        return JSONResponse(
            status_code=exc.http_status,
            content=ApiResponse(code=exc.code, message=exc.message, data=None).model_dump(),
        )

    @app.exception_handler(Exception)
    async def global_handler(request: Request, exc: Exception):
        logger.exception("Unhandled error", request_id=request.state.request_id)
        return JSONResponse(
            status_code=500,
            content=ApiResponse(code=50000, message="internal error", data=None).model_dump(),
        )
```

- service 层抛 `BizError(code, message)`，路由层不处理
- 路由层不抛 `HTTPException`，统一走 `BizError`
- 全局兜底 500，记录 request_id

## 12. 安全规范

### 12.1 API key 管理

- **绝不硬编码** API Key
- `.env` / `.env.production` 必须 `.gitignore`
- 生产用环境变量注入（systemd EnvironmentFile 或 Docker secret）
- 双 Key 独立：`DASHSCOPE_API_KEY` + `BAILIAN_API_KEY`，便于单独轮换
- 定期（每季度）轮换 Key

### 12.2 CORS 白名单

```python
# 开发
CORS_ORIGINS=["*"]

# 生产
CORS_ORIGINS=["https://food-healing.example.edu.cn"]
```

- 生产禁止 `*`，必须具体域名
- `allow_credentials=False`（无 Cookie 鉴权场景）

### 12.3 限流（slowapi）

| 接口 | 限流 | 理由 |
|---|---|---|
| `/api/recognize-dish` | 10/分钟 per IP | VL 调用贵 |
| `/api/chat` | 20/分钟 per user | 防滥用 |
| `/api/mood` | 30/分钟 per user | 高频但便宜 |
| `/api/recommend` | 60/分钟 per user | 纯本地计算 |
| `/api/menu` | 60/分钟 per IP | 缓存命中 |
| 其他 GET | 120/分钟 per IP | 宽限 |
| 其他 POST | 60/分钟 per user | 宽限 |

```python
from slowapi import Limiter
from slowapi.util import get_remote_address

limiter = Limiter(key_func=get_remote_address, storage_uri=settings.REDIS_URL)

@app.post("/api/recognize-dish")
@limiter.limit(settings.RATE_LIMIT_RECOGNIZE)
async def recognize(...): ...
```

- Redis 作为限流存储后端，多 worker 共享计数
- 触发限流返回 429 + 错误码 `42901`

### 12.4 输入校验

- 全部走 Pydantic 模型校验，禁止手工取 `request.json()`
- `extra="forbid"` 拒绝多余字段
- 字符串字段加 `max_length`，防止超长输入
- 枚举字段用 `Literal`，禁止裸 `str`

### 12.5 SQL 注入防护

- ORM（SQLAlchemy）默认参数化，禁止拼接 SQL
- 原生 SQL（PostgreSQL 窗口函数）必须用参数：

```python
# 正确
await conn.fetch(sql, user_id, week_start)

# 错误
await conn.execute(f"SELECT ... WHERE user_id = '{user_id}'")
```

### 12.6 XSS 防护

- 用户输入文本（mood note、comment）落库前不转义，输出时由前端转义
- 后端不返回 HTML，统一返回 JSON
- AI 生成内容若包含 `<script>` 标签，前端渲染时按纯文本处理

### 12.7 文件上传安全

- **类型白名单**：仅允许 `image/jpeg`、`image/png`、`image/webp`
- 校验 magic bytes，拒绝伪装扩展名：

```python
ALLOWED_MAGIC = {
    b"\xff\xd8\xff": "image/jpeg",
    b"\x89PNG": "image/png",
    b"RIFF....WEBP": "image/webp",
}
```

- **大小限制**：5MB（`MAX_IMAGE_SIZE_MB`），超限返回 `40001`
- OSS 用**独立 bucket**：`food-healing-images`，与代码 / 日志隔离
- OSS bucket 开启防盗链（Referer 白名单）

### 14.5 Docker 部署（可选）

部署方式由用户自行选择（systemd 或 Docker Compose）。项目已提供 Docker 支持文件：

- `backend/Dockerfile` — 基于 `python:3.11-slim`，通过 pip 安装 uv（避免 ghcr.io 在国内拉取超时），使用阿里云 PyPI 镜像加速
- `backend/docker-compose.yml` — 编排 web / consumer / scheduler 三个容器
- `backend/.dockerignore` — 排除 `.venv`、`__pycache__`、日志等

Dockerfile 核心要点：

```dockerfile
FROM python:3.11-slim

# 阿里云 PyPI 镜像加速
ENV PIP_INDEX_URL=https://mirrors.aliyun.com/pypi/simple/
ENV UV_INDEX_URL=https://mirrors.aliyun.com/pypi/simple/

# 通过 pip 安装 uv（避免 ghcr.io 在国内拉取超时）
RUN pip install uv --no-cache-dir

WORKDIR /app
COPY pyproject.toml uv.lock ./
RUN uv sync --frozen --no-dev
COPY . .

# 通过 MODE 环境变量切换运行模式：web / consumer / scheduler
CMD ["uv", "run", "uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000", "--workers", "4"]
```

## 15. 提交规范

### 15.1 commit message 格式

```
<type>(<scope>): <subject>

<body>

<footer>
```

**type** 取值：

| type | 含义 |
|---|---|
| feat | 新功能 |
| fix | bug 修复 |
| docs | 文档变更 |
| refactor | 重构（不改行为） |
| perf | 性能优化 |
| test | 测试相关 |
| chore | 构建 / 依赖 / 配置 |
| ci | CI 配置 |

**scope**：模块名，如 `recognize`、`chat`、`habit`、`config`

**示例**：

```
feat(chat): 实现对话短期记忆 Redis List 存储

- 用 LPUSH + LTRIM 保留最近 4 轮
- TTL 1h，超时自动清理
- 每轮对话后异步更新

Closes #123
```

```markdown
## 改动说明
```
