# 食愈校园 — 项目分工实施指南

> 本文档面向**后续执行 agent**，说明项目文档体系、分工方式、阶段划分、协作规范。Agent 接到任务后应先通读本文档，再按角色定位阅读对应的设计/规范文档，最后按阶段推进实施。

## 一、项目概述

### 1.1 产品定位
**"会成长、会记忆、会陪伴的校园饮食伙伴"**——不做游戏化养成（同赛竞品已做），做治愈系陪伴 + 真懂你。AI 搭子有人设（食堂阿姨/学长/学姐）、有长期记忆（记得你说过胃不舒服、对花生过敏、上周蛋白不够）、能承接上下文。

### 1.2 四大产品支柱
| 支柱 | 含义 |
|---|---|
| 看见（识） | 多模态输入：拍照识菜 + 语音 + 外卖截图 + 手动 |
| 懂得（算） | 个性化营养：健康档案、过敏忌口、长期趋势、AI 周报 |
| 陪伴（聊） | 有记忆的 AI 搭子：人设化对话、长期记忆、情绪日记、治愈话语、餐次提醒 |
| 连接（聚） | 校园社交：同学搭配匿名榜、拼饭、菜品评价、食堂今日菜单+推荐 |

### 1.3 技术栈
- **前端**：React 19 + Vite 8 + TailwindCSS 4，三端兼容（手机/平板/网页），构建产物为 `dist/` 静态文件
- **后端**：Python 3.11+ / FastAPI / uv 管理依赖（pyproject.toml + uv.lock）
- **AI**：双平台——通义千问 VL 识图（qwen-vl-max-latest）+ 阿里百练对话（qwen3.7-plus/max/flash 三档路由）
- **基础设施**：PostgreSQL（业务关系 + 时序）+ Redis（缓存/短期记忆/会话）+ RabbitMQ（异步识图 + 消息推送）+ MongoDB（可选，AI 对话历史/识图日志/每日简报）+ MinIO（对象存储）+ Nacos（可选，配置中心）

### 1.4 量级假设
单校单食堂，用户 < 1万。无需微服务，单体 FastAPI 即可。Nacos 主要做配置中心。

## 三、Agent 分工

### 3.1 后端开发 agent
**职责**：实现 FastAPI 后端，包括接口、数据库、AI 调用、异步任务、部署。

**输入**：后端设计文档、后端技术规范、服务器信息。

**输出**：
- `backend/` 目录完整代码（main.py、routes/、services/、data/、config/、ai/、tests/）
- `pyproject.toml` + `uv.lock`
- `.env.example`
- Alembic 迁移脚本
- 单元测试（覆盖率 ≥ 80%）

**关键约束**：
- uv 管理依赖，禁止用 pip install
- AI 模型名集中在 `config.py`，业务代码只传 scene
- Redis key 必须带 `food_healing:` 前缀 + TTL
- 统一响应 `{code, message, data}`，错误码 5 段（1xxxx 业务/2xxxx AI/3xxxx DB/4xxxx 限流鉴权/5xxxx 系统）
- 双 API key 隔离：DASHSCOPE_API_KEY（识图）+ BAILIAN_API_KEY（对话）

### 3.2 前端开发 agent
**职责**：实现 React 19 前端，包括三端兼容 UI、API 调用层、组件、状态管理、PWA、B 端后台。

**输入**：前端设计文档、前端 UI 设计文档、前端技术规范、后端接口契约。

**输出**：
- `frontend-v2/src/` 完整源码（main.jsx、App.jsx、components/、stores/、lib/、hooks/、data/）
- `frontend-v2/dist/` 生产构建产物
- `index.html`（入口 HTML）
- `vite.config.js`（Vite 构建配置 + API 代理）
- `package.json`（npm 依赖声明）

**关键约束**：
- React 19 函数组件 + Hooks，Vite 8 构建，TailwindCSS 4 样式
- 三端断点：768px / 1024px
- 治愈系风格硬约束：CSS 变量优先、禁 `!important`、禁硬编码圆角/间距
- 识图异步轮询：1.5s 间隔 / 20 次 / 30s 超时
- SSE 用 fetch + ReadableStream（非 EventSource，支持 POST）
- 状态管理用 Zustand 5 + persist 中间件，动画用 Framer Motion
- user_habits 后端维护，前端不存明文，仅持久化 allergens/healthProfile/persona 用于离线兜底

### 3.3 联调 agent
**职责**：解决前后端契约不一致，端到端流程跑通。

**输入**：两份设计文档、验收规范 §8.0 前置阻断项清单。

**输出**：
- 契约对齐修正（前后端任一侧改）
- 端到端测试报告（拍照→识别→菜盘→AI 对话→周报→社交）
- 三端联调报告

**首要任务**：解决验收规范 §8.0 列出的 16 处契约不一致（见下方 §六）。

### 3.4 验收 agent
**职责**：照验收规范执行验收，输出验收报告。

**输入**：验收规范、所有交付物。

**输出**：验收报告（markdown，含 PASS/FAIL 清单）。

**关键约束**：
- 性能红线：识图 POST < 200ms 返回 task_id、AI 首 token < 2s、推荐 < 50ms、轮询 < 50ms
- 30s 识图超时硬约束
- uv 强制：`uv sync` 一键复现 + `uv.lock` 提交
- 降级必须实现：SSE 降级、识图失败重试、后端不通 fallback 到本地 generateAiText()

## 四、开发阶段划分

### Phase 0：阻断项解决 + 骨架搭建（1-2 天）

**目标**：拍照识菜 + AI 对话跑通。

**验收**：
- 拍照→识别→加入菜盘 端到端通

## 六、前置阻断项（Phase 0 必须先解决）

验收 agent 通读两份设计文档后发现 **16 处前后端契约不一致**，详见 [验收规范.md §8.0](file:///d:/desktop/Trae/food-healing-demo/docs/验收规范.md)。主要类别：

1. **字段命名风格**：`taskId` vs `task_id`、`userId` vs `user_id`
2. **人设 ID**：`canteen_aunt` vs `auntie`、`senior_brother` vs `senior`
3. **接口路径**：健康档案 `/api/health/profile` vs `/api/profile/health`、运动 `/api/sport/record` vs `/api/exercise`
4. **请求方法**：周报 GET vs POST、分享卡片 GET vs POST
5. **AI 对话请求体结构**：messages 数组格式、stream 字段、user_habits 是嵌入还是引用
6. **响应字段**：confidence vs score、matched vs is_matched

**处理流程**：
1. 联调 agent 逐条核对，选定一种命名（建议统一用 snake_case，符合 Python 后端习惯）
2. 修改前后端设计文档（V2.x 版本）
3. 输出统一契约文档 `docs/接口契约统一.md`（或更新到设计文档）
4. 后续 agent 严格按统一契约实现

## 七、协作规范

### 7.1 接口契约一致性
- 任何接口变更必须同步更新两份设计文档
- 字段命名统一 snake_case（后端 Python 习惯，前端 JS 调用时不强制转 camelCase，保持一致）
- 统一响应 `{code, message, data}`，错误码按 5 段（见后端技术规范 §10）

### 7.2 提交规范
- commit message 格式：`<type>(<scope>): <subject>`，type ∈ feat/fix/docs/refactor/test/chore
- PR 模板：见后端技术规范 §15 / 前端技术规范 §14
- 一次 PR 只做一件事，便于 review

### 7.3 Code Review 清单
- 后端：见后端技术规范 §15
- 前端：见前端技术规范 §16
- 通用：治愈系风格一致性、性能、安全、可访问性

### 7.4 文件操作规范（用户硬约束）
- **文件内容读写必须用内置 Read/Write/Edit 工具，或 Python/Node.js 脚本**
- **禁止用 PowerShell 命令处理文件内容**（cat/sed/awk 等）
- PowerShell 仅用于启动服务、git 等非文件内容操作

## 九、关键决策点（Agent 必须遵守）

### 9.1 性能红线（不可妥协）
| 指标 | 红线 |
|---|---|
| 识图 POST 返回 task_id | < 200ms |
| AI 对话首 token | < 2s |
| 推荐接口 | < 50ms |
| 识图轮询单次 | < 50ms |
| 识图总等待 | ≤ 30s（硬约束） |
| 并发 | 100 QPS |
| 95 分位响应 | < 500ms |

### 9.2 uv 强制
- `pyproject.toml` + `uv.lock` 入 git
- 禁止 `pip install` 与手编 lockfile
- `requirements.txt` 仅作导出产物
- 团队协作：克隆后 `uv sync` 一键复现

### 9.3 降级必须实现
- SSE 失败降级到 `?stream=false`
- 识图失败重试 + 兜底提示
- 后端不通时前端 fallback 到本地 generateAiText()
- AI 模型不可用降级到下一档（max→plus→flash）

### 9.4 安全红线
- API key 不进代码库、不进日志、不进响应
- CORS 白名单（生产只允许前端域名）
- 限流：AI 接口严限（如 10 req/min/user），普通接口宽限
- 文件上传：类型白名单（jpg/png/webp）、大小 ≤ 5MB、OSS 隔离桶

### 9.5 治愈系风格一致性
- CSS 变量优先（`--color-primary` 等）
- 禁 `!important`、禁硬编码圆角/间距
- AI 气泡、主按钮、餐次卡颜色不可破例
- 新增组件必须用 Ma Shan Zheng 标题 + 虚线边框 + 暖色调

## 十二、Agent 协作约定

1. **不越界**：后端 agent 不改前端代码，反之亦然。契约变更走联调 agent。
2. **先读后写**：接到任务先 Read 必读文档，不凭记忆推测。
3. **最小改动**：按用户规则，性能优化先并发→缓存→批处理，从最小改动开始。
4. **问数据量级**：涉及大规模数据操作（全表扫描、批量更新）先问量级。
5. **验证交付**：写完代码后 agent 必须自测对照验收规范，不达标不交付。
6. **报告格式**：每个 agent 完成后返回简短摘要（路径 + 章节/文件 + 关键决策），不超过 250 字。
